# DWH Phase 3C Final Governance Note

Deepwater Horizon 2010 remains a separate `Phase 3C` external rich-data transfer-validation lane in the thesis-facing workflow. It is kept separate from the Mindoro main-case provenance story and is not treated as a second local `Phase 1` calibration study. The thesis-facing DWH lane does not ingest new drifter data and does not inherit the Mindoro or Phase 1 drifter-selected baseline logic. Instead, it keeps the already-authoritative readiness-gated historical forcing stack fixed as `HYCOM GOFS 3.1 currents + ERA5 winds + CMEMS wave/Stokes`, because the DWH objective here is transfer validation on a rich public-observation case rather than reopening forcing-recipe competition.

Truth in Phase 3C comes from the public observation-derived DWH daily masks and their event-corridor union. These masks are used honestly as date-composite products: the initialization composite is `2010-05-20`, and the validation composites are `2010-05-21`, `2010-05-22`, and `2010-05-23`. The repo should not imply exact sub-daily observation acquisition times when those times are not defensible from the stored public products. Those official public observation-derived DWH date-composite masks also remain the scoring reference for every displayed DWH FSS value. Under that governance, `C1` means deterministic external transfer validation, `C2` means ensemble extension and deterministic-vs-ensemble comparison, and `C3` means PyGNOME comparator-only. Thesis-facing interpretation should keep deterministic as the clean baseline transfer-validation result, use `mask_p50` as the preferred probabilistic extension, treat `mask_p90` as support/comparison only, and keep PyGNOME in a comparator-only role rather than as truth.

The thesis-facing DWH figure/package matrix now freezes the time labels explicitly as `24 h = 2010-05-21`, `48 h = 2010-05-22`, `72 h = 2010-05-23`, and `event corridor = 2010-05-21_to_2010-05-23`. It also freezes the product names used across the publication package, curated final-output root, registries, and UI: `observation truth context`, `deterministic footprint overlay`, `mask_p50 overlay`, `mask_p90 overlay`, `mask_p50 footprint overview board`, `mask_p90 footprint overview board`, `mask_p50 and mask_p90 dual-threshold overview board`, `mask_p50 vs mask_p90 vs PyGNOME three-row overview board`, and `PyGNOME footprint overlay`. The thesis-facing order is truth context first, then `C1` deterministic, then `C2` `mask_p50`, then the daily `mask_p50` / `mask_p90` / exact dual-threshold overview boards, then the deterministic-vs-`mask_p50`-vs-`mask_p90` comparison boards, then `C3` PyGNOME-vs-observed, then the daily OpenDrift-vs-PyGNOME overview boards including the three-row `mask_p50` / `mask_p90` / PyGNOME daily board, and finally the OpenDrift-vs-PyGNOME support boards. The current semantics-sync pass treats those `C1/C2/C3` tracks as frozen reportable stored outputs rather than deferred placeholders.
